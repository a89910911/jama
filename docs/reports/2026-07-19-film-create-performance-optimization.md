# FilmCreate 页面性能优化方案

## 1. 结论

`/film/:id?episode=:episodeId` 的卡顿不是单一慢查询造成的，而是三类问题叠加：

1. 页面按分镜逐条加载图片和视频，34 个分镜会产生 68 个媒体列表请求。
2. 页面恢复生成任务时，按剧集、项目、角色、道具、场景分别查询任务；本项目会再产生约 16 个任务请求，而且返回了已经完成任务的超大 `result`/`prompt_snapshot`。
3. 项目详情查询使用 `SELECT *` 读取角色、场景、道具。数据库中部分 `image_url` 是 base64，单字段最高约 12 MB；接口最终虽然会丢弃 data URL，但 SQLite 已先把约 117 MB 的无效数据读入 Node。

本次已完成请求合并、任务过滤、详情查询裁剪和热点索引建设。以当前项目（1 集、34 个有效分镜）为样本：

| 指标 | 优化前 | 优化后 |
| --- | ---: | ---: |
| 首屏媒体 HTTP 请求 | 68 | 1 |
| 首屏任务恢复 HTTP 请求 | 约 16 | 1 |
| 估算首屏业务接口总数 | 约 88 | 约 6 |
| `/dramas/1` 本机稳定耗时 | 约 620–786 ms | 约 107 ms |
| 无进行中任务时的任务恢复响应 | 最高约 68.9 MB | 65 B |
| 单集媒体批量接口耗时 | 不适用（原 68 请求约 39 ms） | 约 1–11 ms |

以上是本地开发环境的代表性采样，不等同于正式压测结果。

## 2. 已实施优化

### 2.1 单集媒体批量加载

新增：

```text
GET /api/v1/episodes/:episode_id/media?storyboard_ids=...
```

后端以两条 SQL 一次性读取该集的图片和视频，并按 `storyboard_id` 分组返回。使用窗口函数限制每个分镜的历史记录：

- 图片最多 100 条/分镜；
- 视频最多 50 条/分镜；
- 忽略软删除的剧集、分镜和媒体记录。

`FilmCreate` 和 `DramaCanvas` 均已切换到批量接口。单条分镜生成完成后的局部刷新仍保留原有单分镜接口，避免不必要的整集刷新。

### 2.2 生成任务批量恢复

任务列表接口新增：

```text
GET /api/v1/tasks?resource_ids=...&active_only=1
```

改动包括：

- 一次请求查询项目、剧集、角色、道具、场景的相关任务；
- 只返回 `pending`、`processing`、`running` 状态；
- 角色与场景使用实际的 `character_<id>`、`scene_<id>` 资源键；
- 按任务类型过滤，避免相同数字 ID 在角色、道具、场景之间串线。

### 2.3 项目详情查询裁剪

角色、场景、道具查询不再使用 `SELECT *`：

- 只选择 API 实际返回的列；
- 在 SQL 层把 `data:` 图片 URL 转为 `NULL`，避免超大 base64 跨越 SQLite/Node 边界；
- 项目级资源只查询一次，再复用于剧集资源组装；
- 剧集角色只查询关联 ID；
- 剧集场景和道具在内存中按 `episode_id`/关联 ID 分组；
- 删除重复的角色、场景、道具全量读取和 `DISTINCT p.*` 大字段排序。

### 2.4 数据库索引

迁移 `28_performance_indexes.sql` 为以下热点路径建立部分索引：

- 项目 → 剧集；
- 剧集 → 分镜；
- 项目/剧集 → 角色、场景、道具；
- 分镜 → 图片/视频历史；
- 项目 + 状态 → 图片/视频生成记录；
- 资源 → 异步任务；
- 状态 → 活跃异步任务。

索引只覆盖 `deleted_at IS NULL` 的有效数据，减少索引体积和无效扫描。

## 3. 后续优化路线

### P1：处理数据库中的 base64 存量

当前数据库约 326 MB，主要风险是图片仍可能以内联 base64 存在业务表中。建议：

1. 先备份数据库；
2. 把 `characters/scenes/props.image_url` 中的 data URL 写入项目文件目录；
3. 数据库只保存 `local_path` 或短 URL；
4. 校验文件完整性和引用关系；
5. 低峰期执行 `VACUUM` 回收空间；
6. 执行 `ANALYZE`/`PRAGMA optimize` 更新查询统计信息。

这属于数据迁移，应单独提供可回滚脚本，不建议在普通启动流程中自动执行。

### P1：拆分项目详情接口

当前 `/dramas/:id` 同时返回项目、所有剧集、分镜和全部资源。项目集数增长后响应仍会线性膨胀。建议拆为：

- `/dramas/:id/summary`：项目与剧集摘要；
- `/episodes/:id/detail`：当前集剧本、角色、场景、道具和分镜；
- `/episodes/:id/media`：当前集媒体历史；
- 其他剧集按切换时懒加载。

目标是首屏只加载当前集，避免大项目一次返回所有集。

### P1：请求生命周期治理

- 同一剧集详情的并发请求做 Promise 合并；
- 路由切集时取消或忽略旧请求；
- 生成任务轮询使用指数退避：前 30 秒 2 秒一次，之后逐步降到 5–10 秒；
- 页面不可见时暂停非关键轮询，恢复可见时立即刷新一次；
- 只在生成任务处于活动状态时启动轮询。

### P2：前端渲染与包体

- 分镜列表超过 50 条时使用虚拟列表；
- 图片历史、视频历史在对应区块进入视口后再加载；
- 缩略图统一生成小尺寸版本，列表不加载原图；
- 按 FilmCreate、DramaCanvas、AI 配置继续拆分共享大依赖。当前构建仍提示主 chunk 超过 500 kB；
- 对静态组件使用异步加载，减少首屏解析和组件初始化开销。

### P2：可观测性与性能预算

建议增加：

- 请求日志：路径、状态码、耗时、响应字节数；
- SQLite 慢查询日志：默认阈值 50 ms；
- 页面指标：FCP、LCP、INP、长任务数量；
- 首屏请求预算：普通打开不超过 10 个业务接口；
- 响应预算：项目详情 P95 < 200 ms，媒体批量接口 P95 < 100 ms；
- 自动回归：测试固定 100 分镜、每分镜 20 张图/5 个视频的样本项目。

## 4. 验收建议

1. 无进行中任务时打开页面，任务状态接口只请求一次且返回空数组。
2. 34 个分镜只出现一次 `/episodes/1/media`，不再出现 68 个 `/images`/`/videos`。
3. 切换剧集时只加载目标集详情和媒体。
4. 生成单个分镜图/视频完成后，仅刷新该分镜媒体。
5. 角色、场景、道具中的 data URL 不出现在项目详情响应中。
6. 后端全量测试、前端全量测试和生产构建均通过。
